Bugs encontrados y corregidos
No se encontraron bugs en el codigo de produccion en esta fase. Los ajustes realizados fueron exclusivamente en la infraestructura de cleanup de datos de prueba.

Ajuste en cleanupUsers — columnas created_by en tablas de plantas

Las tablas employees y fields tienen columnas created_by que referencian a users. Al eliminar los usuarios de prueba, estas FKs debian nulificarse primero para no violar restricciones de integridad referencial.

// Agregado a cleanupUsers() para el modulo de plantas
await fieldRepo.query(`UPDATE employees SET created_by = NULL WHERE created_by IN (${ph})`, ids);
await fieldRepo.query(`UPDATE fields SET created_by = NULL WHERE created_by IN (${ph})`, ids);

Vulnerabilidades documentadas
Los siguientes hallazgos estan documentados en los tests pero requieren evaluacion del equipo antes de corregirse.
MEDIO Coordinator puede crear y editar campos y empleados, pero no eliminarlos
Descripcion: El rol coordinator tiene acceso de escritura a campos y empleados (POST/PATCH), pero no de eliminacion (DELETE). Este comportamiento es intencional segun el diseno del sistema, pero queda documentado como asimetria de permisos que puede causar confusion.

Test que lo documenta: [SEGURIDAD] Coordinator NO puede eliminar campos
src/plants/fields/fields.controller.ts — metodo remove()
MEDIO Cualquier usuario autenticado puede listar y ver campos y empleados
Descripcion: Los endpoints GET de campos y empleados no restringen por rol. Un usuario recien creado sin rol asignado puede obtener el listado completo de plantas y su personal.

Impacto: Exposicion de estructura organizacional, localizaciones de campo y datos de empleados a cualquier usuario que tenga credenciales validas en el sistema, independientemente de su nivel de acceso.

Correccion sugerida: Evaluar si las lecturas deben requerir al menos el rol coordinator o si el comportamiento actual es intencional.
src/plants/fields/fields.controller.ts, src/plants/employees/employees.controller.ts

Patron tecnico establecido para modulos de plantas

Usuarios de prueba con campo asignado

Algunas operaciones (crear cuadrillas, registrar en bitacora) requieren que el usuario autenticado tenga un field_id asignado. La estrategia consiste en crear el campo primero y luego asignarlo al usuario en el beforeAll:

// Crear campo de prueba primero
const testField = await fieldRepo.save(
  fieldRepo.create({ name: 'QA-Plants-Field', location: 'QA Location' }),
);
// Asignar al usuario admin/supervisor
adminUser = await userRepo.save(userRepo.create({
  email: ADMIN_EMAIL, password_hash: hash,
  field_id: testField.id,  // campo requerido para operaciones de planta
  ...
}));

Prefijo QA en todos los recursos de prueba

Todos los recursos creados durante los tests usan el prefijo QA- en nombres y qa- en emails. Esto permite limpiar datos residuales por patron sin afectar datos de produccion:

// Limpieza de campos de prueba en beforeAll
const staleField = await fieldRepo.findOne({ where: { name: 'QA-Plants-Field' } });
if (staleField) await cleanupField(staleField.id);
