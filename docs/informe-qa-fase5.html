Resumen ejecutivo

Se implementaron 30 tests E2E para el modulo de Consumables cubriendo el catalogo de insumos, los periodos contables, solicitudes de planta y requisiciones formales. Esta fase revelo el hallazgo de seguridad mas grave de todo el proyecto: el controlador de insumos no tiene ningun guard de autenticacion ni autorizacion. Adicionalmente, varios endpoints de solicitudes y requisiciones son publicos sin justificacion de diseno clara.

Atencion: InsumoController expone operaciones de escritura (crear, modificar, eliminar, cerrar periodo) sin autenticacion. Es la unica entidad del sistema completamente desprotegida.

Insumos sin auth

POST, PATCH y DELETE de insumos no requieren ningun token. Cualquier cliente HTTP anonimo puede modificar el catalogo.

mis-solicitudes puede ser 404

Si el usuario autenticado no tiene field_id asignado, el servicio retorna 404, no 200 con array vacio.

recepcion si requiere auth

Solo PATCH /requisiciones/:id/recepcion tiene guard. El resto del controlador de requisiciones es publico.

Bug encontrado y corregido
Bug 1 — GET mis-solicitudes retorna 404 cuando el usuario no tiene campo asignado
Archivo afectado: test/consumables.e2e-spec.ts

Causa: El servicio findMisSolicitudes busca por el field_id del usuario autenticado. Si el usuario no tiene campo asignado, el servicio lanza un NotFoundException y retorna 404. El test asumia que retornaria 200 con un array vacio.

Nota: Este es el comportamiento correcto del servicio (no es un bug del API). La asuncion incorrecta estaba en el test. 
// ANTES — asumia siempre 200
const res = await request(app).get('/api/v1/solicitudes/mis-solicitudes?mes=11&anio=2099')
  .set('Authorization', `Bearer ${adminToken}`);
expect(res.status).toBe(200);
expect(Array.isArray(res.body)).toBe(true);

// DESPUES — acepta 200 o 404 segun si el usuario tiene campo
expect([200, 404]).toContain(res.status);
if (res.status === 200) {
  expect(Array.isArray(res.body)).toBe(true);
}

Vulnerabilidades de seguridad documentadas
Esta es la fase con mayor deuda de seguridad del proyecto. El modulo de insumos es el unico controlador sin ninguna forma de autenticacion en toda la aplicacion. La correccion debe ser prioritaria antes de cualquier despliegue en produccion.
CRITICA InsumoController sin JwtAuthGuard en ningun endpoint
Descripcion: El controlador completo de insumos (InsumoController) no aplica @UseGuards(JwtAuthGuard) ni a nivel de clase ni en ningun metodo. Todos los endpoints de escritura estan completamente abiertos al publico.

Endpoints afectados:
  • POST /api/v1/insumos — crear insumo (sin auth)
  • PATCH /api/v1/insumos/:id — actualizar precio/datos (sin auth)
  • DELETE /api/v1/insumos/:id — eliminar del catalogo (sin auth)
  • POST /api/v1/insumos/cerrar-mes — cerrar periodo contable (sin auth)
  • PATCH /api/v1/insumos/:id/borrador — crear borrador de precio (sin auth)
Vector de ataque: Cualquier script o herramienta HTTP puede vaciar el catalogo de insumos, modificar todos los precios o cerrar el periodo contable sin ninguna credencial.

Correccion requerida: Agregar @UseGuards(JwtAuthGuard, RolesGuard) a la clase InsumoController y decorar cada metodo con los roles apropiados (@Roles('admin', 'coordinator') para escritura, opcionalmente sin restriccion de rol para lectura).

ALTO Solicitudes y Requisiciones: endpoints de lista y creacion publicos
Descripcion: Los endpoints de listado (GET /solicitudes, GET /requisiciones, GET /requisiciones/informe) y potencialmente los de creacion estan disponibles sin autenticacion.

Datos expuestos: Volumen de solicitudes por campo, valores estimados de compras, presupuestos mensuales, identificadores de campos y periodos contables.

Posible justificacion: Podria ser intencional si se requiere que paneles de gestion externos (sin login) puedan visualizar el estado de solicitudes. Si es asi, debe documentarse explicitamente en el codigo como decision de diseno.

MEDIO GET /insumos/borradores expone precios pendientes de aprobacion
Descripcion: Los borradores de precios son propuestas de cambio antes de cerrar el periodo. Exponer esta informacion publicamente permite a cualquier proveedor conocer los precios que la empresa esta evaluando antes de que sean oficiales, lo que puede afectar las negociaciones.

Correccion sugerida: Restringir GET /insumos/borradores a roles con permiso de gestion de precios.
src/consumables/insumos/insumo.controller.ts — getBorradores()
MEDIO cerrar-mes publico permite cierre no autorizado del periodo contable
Descripcion: El endpoint POST /api/v1/insumos/cerrar-mes no requiere autenticacion. Cerrar un periodo contable es una operacion irreversible que congela los precios y las solicitudes del mes. Un actor externo malicioso podria cerrar el periodo antes de tiempo, bloqueando solicitudes pendientes.

Correccion requerida: Restringir exclusivamente a admin o al rol de finanzas/contabilidad con confirmacion adicional.

Patron tecnico establecido

Limpieza de insumos con prefijo QA

Los insumos de prueba usan el prefijo QA-CONS- en la descripcion para identificarlos y limpiarlos de forma segura:

async function cleanupTestInsumos() {
  await repo.query(`DELETE FROM insumos_borradores WHERE insumo_id IN (SELECT id FROM insumos WHERE descripcion LIKE 'QA-CONS-%')`);
  await repo.query(`DELETE FROM insumos_historial WHERE insumo_id IN (SELECT id FROM insumos WHERE descripcion LIKE 'QA-CONS-%')`);
  await repo.query(`DELETE FROM insumos WHERE descripcion LIKE 'QA-CONS-%'`);
}

Tests de endpoints sin auth: documentar el comportamiento esperado

Para endpoints que NO deberian ser publicos pero lo son, el test documenta la vulnerabilidad con el prefijo [SEGURIDAD] y verifica que el endpoint funciona sin token. Esto crea un registro del comportamiento actual que activara una falla si el endpoint se protege en el futuro (lo cual seria la correccion correcta):

// El test confirma que la vulnerabilidad existe y es reproducible
it('[SEGURIDAD] crear insumo no requiere autenticacion — endpoint sin proteccion', async () => {
  const res = await request(app.getHttpServer())
    .post('/api/v1/insumos')
    // Sin .set('Authorization', ...)
    .send({ descripcion: INSUMO_DESC, unidad: 'unidad', categoria: 'PAPELERIA' })
    .expect(201); // Cuando se corrija, aqui deberia esperarse 401
});
