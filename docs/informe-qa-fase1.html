Resumen ejecutivo

Se realizo una revision completa de seguridad y pruebas automatizadas end-to-end sobre los modulos de autenticacion, usuarios y roles del sistema. El objetivo fue identificar vulnerabilidades, validar el correcto funcionamiento de todas las rutas API y establecer un entorno de testing reproducible antes de continuar con el desarrollo de nuevas funcionalidades.

Resultado final: 68 de 68 tests pasando en dos corridas consecutivas sobre base de datos de prueba aislada (sgoc_test_db). El entorno es idempotente: puede ejecutarse multiples veces sin dejar datos sucios.

Modulos cubiertos

Auth (login, refresh, logout), Users (CRUD completo, cambio de clave, reset), Roles (CRUD, permisos, restriccion de sistema).

Tipos de test

Funcionales (flujo feliz), de seguridad (RBAC, JWT, inyeccion), y negativos (400/401/403/404/409).

Infra de prueba

BD aislada sgoc_test_db, servicios externos mockeados (SendGrid), limpieza automatica pre y post suite.

Bugs encontrados y corregidos
Los siguientes 6 problemas fueron identificados durante la fase de testing y corregidos en el mismo proceso.
Bug 1 (CRITICO) — CreateRoleDto.description sin decoradores de validacion
Archivo: src/roles/dto/create-role.dto.ts

Comportamiento erroneo: Todo POST /roles devolvía 400, incluso con payload completamente correcto. Con forbidNonWhitelisted: true y transform: true, class-transformer instancia el DTO con description como propiedad propia aunque sea undefined. class-validator rechaza toda propiedad sin decorador.

Impacto en produccion: Creacion de roles 100% bloqueada si se enviaba el campo description (o incluso si el middleware lo interpretaba como presente). Bug silencioso que solo se detecta con tests.

Fix aplicado: 
// ANTES — description sin decoradores
export class CreateRoleDto {
  @IsString() name!: string;
  @IsString() slug!: string;
  description?: string; // sin @IsOptional() ni @IsString()
}

// DESPUES — corregido
export class CreateRoleDto {
  @IsString() name!: string;
  @IsString() slug!: string;
  @IsOptional() @IsString() description?: string;
}
Bug 2 — Refresh token sin jti (JWT ID) generaba duplicate key en sessions
Archivo: src/auth/auth.service.ts

Comportamiento erroneo: Si se hacian dos logins en el mismo segundo, el refresh token generado podia tener el mismo hash, causando un conflicto de clave unica en la tabla sessions.

Fix aplicado: Se agrego jwtid: crypto.randomUUID() al firmar el refresh token, garantizando unicidad aunque los payloads sean identicos y ocurran en el mismo segundo. 
// ANTES
const refreshToken = this.jwtService.sign(payload, {
  secret: this.config.get('JWT_REFRESH_SECRET'),
  expiresIn: this.config.get('JWT_REFRESH_EXPIRES_IN'),
});

// DESPUES
const refreshToken = this.jwtService.sign(payload, {
  secret: this.config.get('JWT_REFRESH_SECRET'),
  expiresIn: this.config.get('JWT_REFRESH_EXPIRES_IN'),
  jwtid: crypto.randomUUID(), // garantiza unicidad
});
Bug 3 (SEGURIDAD) — SSL sin rejectUnauthorized en produccion
Archivo: src/app.module.ts

Comportamiento erroneo: La conexion SSL a PostgreSQL tenia rejectUnauthorized: false, lo que permite ataques man-in-the-middle ya que no valida el certificado del servidor de base de datos.

Fix aplicado: Cambiado a rejectUnauthorized: true en produccion.
Bug 4 (SEGURIDAD) — CORS abierto con wildcard *
Archivo: src/main.ts

Comportamiento erroneo: El servidor aceptaba peticiones CORS desde cualquier origen (origin: '*'), permitiendo que cualquier sitio web externo hiciera requests en nombre de usuarios autenticados.

Fix aplicado: Reemplazado por lista explicita de origenes permitidos configurables por variable de entorno.
Bug 5 (SEGURIDAD) — Sin HTTP security headers
Archivo: src/main.ts

Comportamiento erroneo: La API no enviaba headers de seguridad HTTP (X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, etc.), dejandola expuesta a ataques de clickjacking, MIME sniffing y otros.

Fix aplicado: Agregado app.use(helmet()) al bootstrap de la aplicacion.
Bug 6 (SEGURIDAD) — Sin rate limiting en endpoints criticos
Archivo: src/app.module.ts

Comportamiento erroneo: No habia ninguna proteccion contra brute force en los endpoints de autenticacion (POST /auth/login, etc.), permitiendo intentos ilimitados de adivinanza de credenciales.

Fix aplicado: Integrado ThrottlerModule con limite configurable via variable de entorno (THROTTLE_LIMIT). En tests se usa 10000 para no interferir; en produccion se reduce segun politica.

Vulnerabilidades documentadas pendientes de correccion
Los siguientes hallazgos estan documentados en los tests pero NO fueron corregidos en esta fase. Cada uno tiene un test que registra el comportamiento actual y la advertencia correspondiente.
ALTO PATCH /users/:id no tiene restriccion de rol
Descripcion: El endpoint PATCH /api/v1/users/:id no tiene el decorador @Roles() en el metodo ni en la clase del controlador. Cualquier usuario autenticado (sin importar su rol) puede modificar los datos de cualquier otro usuario, incluyendo campos sensibles como position, module, field_id, etc.

Vector de ataque: Usuario sin rol autentica con sus credenciales y hace PATCH /api/v1/users/{id_admin} con cualquier body. Recibe 200.

Correccion sugerida: Agregar @Roles('admin') al metodo update() en UsersController o, si coordinadores tambien deben poder editar, @Roles('admin', 'coordinator').
src/users/users.controller.ts — metodo update()
MEDIO GET /users no tiene restriccion de rol
Descripcion: GET /api/v1/users devuelve el listado paginado de usuarios para cualquier usuario autenticado, independientemente de su rol. Un usuario recien creado sin rol asignado puede ver todos los usuarios del sistema.

Correccion sugerida: Evaluar si el listado de usuarios debe estar restringido a admin y coordinator. Si los operadores de campo no necesitan ver la lista de usuarios, agregar @Roles('admin', 'coordinator').
src/users/users.controller.ts — metodo findAll()
MEDIO JWT_SECRET con entropia insuficiente
Descripcion: El JWT_SECRET configurado tiene formato UUID (xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx), que tiene solo 122 bits de entropia efectiva y un espacio de caracteres limitado. Para un secreto HMAC-SHA256 se recomiendan al menos 256 bits de entropia verdadera.

Correccion sugerida: Regenerar usando node -e "console.log(require('crypto').randomBytes(64).toString('hex'))" y actualizar en el servidor de produccion.
.env — variable JWT_SECRET
BAJO ADMIN_PASSWORD debil en seed de produccion
Descripcion: El valor por defecto de ADMIN_PASSWORD en el seed es Admin123, una contrasena que aparece en todos los diccionarios de brute force conocidos.

Correccion sugerida: Cambiar el valor por defecto a algo aleatorio o, mejor, no tener valor por defecto y requerir que ADMIN_PASSWORD este definida en el entorno de produccion. Verificar que el admin de produccion ya cambio su clave.
src/database/seeds/05-admin.seed.ts
INFO Sin validacion de JWT audience e issuer
Descripcion: Los tokens JWT no incluyen ni validan los claims aud (audience) e iss (issuer). En arquitecturas con multiples servicios, un token generado para un servicio podria ser reutilizado en otro que comparta el mismo secreto.

Correccion sugerida: Agregar audience e issuer al firmar y validar los tokens en JwtModule.registerAsync(). Impacto bajo en arquitectura de un solo servicio, pero importante si se escala.
src/auth/auth.module.ts

Patron tecnico establecido para nuevos tests

Se establecio un patron reproducible para escribir tests E2E de cualquier modulo. Este patron resuelve los problemas encontrados durante el desarrollo de los tests de esta fase:

Problema: FK violations al limpiar datos de prueba

Las columnas created_by en las tablas users, user_roles y roles son FKs nullable sin onDelete: CASCADE. Borrar un usuario que fue created_by de otros registros viola la FK. La funcion cleanupUsers resuelve esto nulificando las FKs antes de borrar:

async function cleanupUsers(ids: string[]) {
  if (ids.length === 0) return;
  const ph = ids.map((_, i) => `$${i + 1}`).join(', ');
  // 1. Nulificar FKs created_by en tablas relacionadas
  await userRoleRepo.query(`UPDATE user_roles SET created_by = NULL WHERE created_by IN (${ph})`, ids);
  await userRepo.query(`UPDATE users SET created_by = NULL WHERE created_by IN (${ph})`, ids);
  await roleRepo.query(`UPDATE roles SET created_by = NULL WHERE created_by IN (${ph})`, ids);
  // 2. Borrar dependientes y luego los usuarios
  await sessionRepo.query(`DELETE FROM sessions WHERE user_id IN (${ph})`, ids);
  await userRoleRepo.query(`DELETE FROM user_roles WHERE user_id IN (${ph})`, ids);
  await userRepo.query(`DELETE FROM users WHERE id IN (${ph})`, ids);
}

Problema: Tests no idempotentes (fallan en segunda corrida)

Si el proceso termina abruptamente con --forceExit, los recursos creados durante los tests (usuarios, roles) quedan en la BD. El beforeAll limpia tanto los usuarios base como los recursos que podrian haber quedado de corridas anteriores:

// En beforeAll — limpiar TODOS los recursos posibles de corridas anteriores
const staleUsers = await userRepo.find({
  where: [
    { email: ADMIN_EMAIL }, { email: COORD_EMAIL },
    { email: PLAIN_EMAIL }, { email: 'qa-nuevo@test.com' }, // creado en POST /users test
  ],
});
await cleanupUsers(staleUsers.map(u => u.id));

const staleRoles = await roleRepo
  .createQueryBuilder('r')
  .where("r.slug LIKE 'qa-%'") // prefijo de roles de prueba
  .getMany();
for (const r of staleRoles) { await roleRepo.delete({ id: r.id }); }

Problema: Mail service hace llamadas reales a SendGrid

El constructor de MailService llama a setApiKey() inmediatamente, pero el mock debe ponerse despues de app.init():

await app.init(); // primero inicializar

const mailService = app.get(MailService);
jest.spyOn(mailService, 'sendVerificationEmail').mockResolvedValue(undefined);
jest.spyOn(mailService, 'sendPasswordResetByAdmin').mockResolvedValue(undefined);
Regla de nomenclatura: Todos los recursos de prueba usan el prefijo qa-. Emails: qa-*@test.com. Slugs de roles: qa-*. Esto permite limpiarlos por patron sin afectar datos de produccion o desarrollo.
