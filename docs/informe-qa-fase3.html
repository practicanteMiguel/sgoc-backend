Bugs encontrados y corregidos
Se encontraron y corrigieron 3 bugs durante la escritura de los tests. Todos eran asunciones incorrectas del test sobre el comportamiento del API, no bugs del codigo de produccion.
Bug 1 — via_reports no tiene columna field_id directa
Archivo afectado: test/plants.e2e-spec.ts — funcion cleanupField()

Error: QueryFailedError: no existe la columna "field_id" en la relacion "via_reports"

Causa: La tabla via_reports referencia a via_monthly_logs via monthly_log_id. No tiene columna field_id propia. El cleanup intentaba hacer DELETE FROM via_reports WHERE field_id = $1 directamente. 
// ANTES (incorrecto)
DELETE FROM via_report_items WHERE report_id IN (SELECT id FROM via_reports WHERE field_id = $1)
DELETE FROM via_reports WHERE field_id = $1

// DESPUES (cascada en 3 niveles)
DELETE FROM via_report_items WHERE report_id IN (
  SELECT id FROM via_reports WHERE monthly_log_id IN (
    SELECT id FROM via_monthly_logs WHERE field_id = $1))
DELETE FROM via_reports WHERE monthly_log_id IN (
  SELECT id FROM via_monthly_logs WHERE field_id = $1)
DELETE FROM via_monthly_logs WHERE field_id = $1
Bug 2 — La respuesta usa vault_token, no token
Archivo afectado: test/plants.e2e-spec.ts — test de GET vault_token de via-log

Error: Expected path: "token" — Received object: { "vault_token": "..." }

Causa: La entidad ViaMonthlyLog declara la columna como vault_token (@Column({ name: 'vault_token', ... })). El test asumia que la propiedad en la respuesta seria token. 
expect(res.body).toHaveProperty('token');
expect(res.body).toHaveProperty('vault_token');
Bug 3 — CreateRqDesdeDotacionDto.numero_rq es number, no string
Archivo afectado: test/dotaciones.e2e-spec.ts — test de POST /solicitudes/:id/rq

Error: El test esperaba 404 (solicitud no encontrada) pero recibia 400 (validacion del DTO).

Causa: El DTO tiene @IsNumber() numero_rq y @IsArray() @ArrayMinSize(1) items. El test enviaba numero_rq: 'RQ-TEST-001' (string) y omitia items. La validacion del DTO fallaba antes de que el servicio buscara la solicitud. 
// ANTES — DTO invalido, nunca llega al servicio
.send({ numero_rq: 'RQ-TEST-001', lote: 'L01' })
.expect(404);

// DESPUES — DTO valido para que el servicio pueda hacer el lookup
.send({
  numero_rq: 1,
  items: [{ descripcion: 'Item test', unidad: 'unidad',
            tipo_requisicion: 'ORDINARIA', solicitado: 1 }],
})
.expect(404);

Vulnerabilidades documentadas
Los endpoints publicos de dotaciones son diseno intencional para el flujo del supervisor en campo. Se documentan igualmente por el riesgo que representan si el token se compromete.
ALTO dotacion_spaces.supervisor_id no tiene onDelete — FK silenciosa
Descripcion: La entidad DotacionSpace define supervisor_id como FK a users sin onDelete especificado. En la base de datos esto resulta en el comportamiento por defecto de PostgreSQL (NO ACTION), lo que equivale a RESTRICT. Eliminar un supervisor sin borrar sus espacios primero generara un error FK en produccion.

Hallazgo durante testing: El cleanup del test tuvo que eliminar manualmente la cascada dotacion_imagenes → reposiciones_dotacion → solicitudes_dotacion → dotacion_spaces antes de poder eliminar los usuarios supervisores.

Correccion sugerida: Agregar onDelete: 'SET NULL' o onDelete: 'CASCADE' a la relacion supervisor_id en la entidad, segun el comportamiento deseado cuando se da de baja a un supervisor.
src/plants/dotaciones/entities/dotacion-space.entity.ts — supervisor_id FK
MEDIO PATCH /dotaciones/solicitudes/:id/estado sin autenticacion
Descripcion: El endpoint de actualizacion de estado de una solicitud de dotacion (PATCH /dotaciones/solicitudes/:id/estado) no requiere autenticacion. Cualquier actor anonimo que conozca el UUID de una solicitud puede cambiar su estado (ej: de emitida a autorizada).

Diseno intencional vs. riesgo: Segun el flujo de negocio, este endpoint es para que el autorizador en campo (sin VPN) pueda aprobar solicitudes. Sin embargo, sin ninguna forma de validar la identidad del autorizador, cualquier atacante con conocimiento del UUID puede manipular estados de aprobacion.

Correccion sugerida: Agregar al menos validacion por PIN del autorizador, o un token de autorizacion de corta duracion enviado al email del autorizador.

Patron tecnico establecido

Cleanup de cascada multi-nivel para dotaciones

La cadena de FK en dotaciones requiere eliminar en orden inverso estricto antes de poder eliminar el campo o el usuario supervisor. Este patron quedara establecido para cualquier entidad con cascadas profundas:

async function cleanupField(fieldId: string) {
  // Nivel 4: imagenes
  await repo.query(`DELETE FROM dotacion_imagenes WHERE reposicion_id IN (
    SELECT id FROM reposiciones_dotacion WHERE solicitud_id IN (
      SELECT id FROM solicitudes_dotacion WHERE space_id IN (
        SELECT id FROM dotacion_spaces WHERE field_id = $1)))`, [fieldId]);
  // Nivel 3: reposiciones
  await repo.query(`DELETE FROM reposiciones_dotacion WHERE solicitud_id IN (
    SELECT id FROM solicitudes_dotacion WHERE space_id IN (
      SELECT id FROM dotacion_spaces WHERE field_id = $1))`, [fieldId]);
  // Nivel 2: solicitudes
  await repo.query(`DELETE FROM solicitudes_dotacion WHERE space_id IN (
    SELECT id FROM dotacion_spaces WHERE field_id = $1)`, [fieldId]);
  // Nivel 1: espacios
  await repo.query(`DELETE FROM dotacion_spaces WHERE field_id = $1`, [fieldId]);
  // Campo
  await repo.query(`DELETE FROM fields WHERE id = $1`, [fieldId]);
}

Capturar vault_token tras POST

Para tests que dependen del token publico, capturar el vault_token en el beforeAll o en el primer test del grupo y reutilizarlo:

let spaceToken: string;

// En el test que crea el espacio
const res = await request(app.getHttpServer())
  .post('/api/v1/dotaciones/spaces')
  .set('Authorization', `Bearer ${superToken}`)
  .expect(201);
spaceToken = res.body.vault_token; // capturado para tests siguientes
