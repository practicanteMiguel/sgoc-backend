Resumen ejecutivo

Se implementaron 43 tests E2E para el modulo de Compliance cubriendo la generacion de entregables mensuales, CRUD de deliverables, horarios, evidencias en Drive y los cuatro formatos de datos (taxi, pernoctacion, disponibilidad, horas extra). Esta fue la fase con mayor numero de bugs encontrados (5), todos relacionados con asunciones incorrectas sobre la estructura de respuesta del API y el comportamiento de las FKs en la limpieza de datos.

Resultado final: 43 de 43 tests pasando sobre compliance.e2e-spec.ts.

generate-month

Crea los 6 entregables del mes en orden: taxi, pernoctacion, horas_extra, disponibilidad, schedule_6x6, schedule_5x2. Retorna {message, deliverables: []}.

Format type validation

Cada endpoint de formato (taxi, pernoctacion, etc.) valida que el deliverable.format_type coincida con la ruta. Usar el ID incorrecto devuelve 400.

supervisor_id RESTRICT

deliverables y schedules tienen FK supervisor_id con onDelete: RESTRICT. El cleanup debe eliminar deliverables y schedules antes que los usuarios.

Bugs encontrados y corregidos
Se encontraron y corrigieron 5 bugs. Todos eran asunciones incorrectas del test sobre respuestas del API o nombres de columnas en la base de datos.
Bug 1 — deliverables.supervisor_id con onDelete RESTRICT en cleanup
Error: violates foreign key constraint "FK_..." on table "deliverables": update or delete on table "users" violates RESTRICT

Causa: La entidad Deliverable usa supervisor_id FK con onDelete: 'RESTRICT'. El cleanup intentaba actualizar una columna created_by que no existe. Debia eliminar los deliverables por supervisor antes de eliminar el usuario. 
// ANTES (incorrecto — columna created_by no existe en deliverables)
UPDATE deliverables SET created_by = NULL WHERE created_by IN (...)

// DESPUES (eliminar registros hijos primero, luego deliverables)
DELETE FROM taxi_records WHERE deliverable_id IN (SELECT id FROM deliverables WHERE supervisor_id IN (...))
DELETE FROM pernoctacion_records WHERE deliverable_id IN (SELECT id FROM deliverables WHERE supervisor_id IN (...))
DELETE FROM disponibilidad_records WHERE deliverable_id IN (...)
DELETE FROM horas_extra_records WHERE deliverable_id IN (...)
DELETE FROM deliverables WHERE supervisor_id IN (...)
Bug 2 — schedules.supervisor_id tambien con onDelete RESTRICT
Causa: La entidad Schedule tiene el mismo patron que Deliverable: supervisor_id FK con onDelete: 'RESTRICT'. El cleanup tampoco eliminaba schedules antes de los usuarios. 
UPDATE schedules SET created_by = NULL WHERE created_by IN (...)

DELETE FROM schedule_days WHERE schedule_id IN (SELECT id FROM schedules WHERE supervisor_id IN (...))
DELETE FROM schedules WHERE supervisor_id IN (...)
Bug 3 — generate-month retorna {message, deliverables} no un array
Error: deliverableId quedaba undefined y todos los tests de formato fallaban en cascada.

Causa: El servicio retorna { message: "...", deliverables: [...] }. El test asumia que el body era directamente el array de entregables y lo verificaba con Array.isArray(genRes.body). 
// ANTES — Array.isArray(genRes.body) era false siempre
const delivs = Array.isArray(genRes.body) ? genRes.body : [];
deliverableId = delivs[0]?.id;

// DESPUES — extraer del objeto o del array segun el caso
const delivs = [200,201].includes(genRes.status)
  ? (Array.isArray(genRes.body) ? genRes.body : genRes.body?.deliverables ?? [])
  : [];
deliverableId    = delivs.find(d => d.format_type === 'taxi')?.id;
pernoctacionId   = delivs.find(d => d.format_type === 'pernoctacion')?.id;
disponibilidadId = delivs.find(d => d.format_type === 'disponibilidad')?.id;
horasExtraId     = delivs.find(d => d.format_type === 'horas_extra')?.id;
Bug 4 — GET pernoctacion/disponibilidad/horas-extra retornaba 400 con deliverable de tipo taxi
Causa: El servicio valida que deliverable.format_type coincida con el endpoint llamado. Todos los tests de formato usaban el ID del primer deliverable generado (taxi). Llamar GET /deliverables/{taxi-id}/pernoctacion retorna 400 porque el tipo no coincide.

Patron descubierto: REQUIRED_FORMATS = [TAXI, PERNOCTACION, HORAS_EXTRA, DISPONIBILIDAD, SCHEDULE_6X6, SCHEDULE_5X2] — el orden de generacion determina que el indice 0 es siempre taxi, 1 pernoctacion, etc. 
// ANTES — todos usaban el mismo ID (taxi)
GET /compliance/deliverables/${deliverableId}/pernoctacion

// DESPUES — cada formato usa el ID de su propio tipo
GET /compliance/deliverables/${pernoctacionId}/pernoctacion
GET /compliance/deliverables/${disponibilidadId}/disponibilidad
GET /compliance/deliverables/${horasExtraId}/horas-extra
Bug 5 — reminders/trigger retorna 201 y la clave es notifications_sent
Error 1: expected 200 "OK", got 201 "Created" — NestJS asigna HTTP 201 a @Post() por defecto.
Error 2: Expected path: "sent" — Received: { "notifications_sent": [], "deadline_date": "..." } 
.expect(200);
expect(res.body).toHaveProperty('sent');

expect([200, 201]).toContain(res.status);

Patron tecnico establecido

Capturar IDs por format_type en beforeAll

Cuando el setup genera multiples recursos del mismo tipo con variantes (como los 6 deliverables de diferentes formatos), usar .find() por propiedad discriminante en lugar de asumir posicion en el array:

let deliverableId: string;    // taxi
let pernoctacionId: string;
let disponibilidadId: string;
let horasExtraId: string;

// En beforeAll, tras generar:
deliverableId    = delivs.find(d => d.format_type === 'taxi')?.id;
pernoctacionId   = delivs.find(d => d.format_type === 'pernoctacion')?.id;
disponibilidadId = delivs.find(d => d.format_type === 'disponibilidad')?.id;
horasExtraId     = delivs.find(d => d.format_type === 'horas_extra')?.id;

FKs con RESTRICT requieren cleanup activo

Siempre que una FK use onDelete: 'RESTRICT' y el test deba eliminar el registro referenciado, el cleanup debe:

  • Eliminar registros hijo (registros de formatos: taxi_records, pernoctacion_records, etc.)
  • Eliminar el registro padre (deliverable, schedule)
  • Solo entonces eliminar el usuario al que apunta supervisor_id
